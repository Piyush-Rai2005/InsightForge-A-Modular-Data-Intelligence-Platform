InsightForge — Interview-Ready Technical Guide

A Modular Data Intelligence Platform — sourced directly from the actual codebase, including all recent updates.

One-Liner

InsightForge is a full-stack data intelligence platform where users upload any dataset (CSV, Excel, JSON, XML, YAML, Parquet, SQLite, ZIP), and a 10-agent AI pipeline automatically performs data health checks, cleaning, schema analysis, ML model training, generates interactive Plotly visualizations, and produces a PDF report — all exposed via a FastAPI backend with JWT auth, a React + Vite frontend, a natural-language chat engine powered by DuckDB + Groq AI, a threshold-aware RAG layer (ChromaDB + MiniLM), and a geographic synonym system (US Census Bureau divisions) for precise regional queries.

The 10-Agent ML Pipeline

Each agent extends BaseAgent with run(context) → context. A shared context dict accumulates state as it flows through all 10 agents. Non-critical agents are wrapped in try/except — pipeline continues on failure. Agents 3, 4, and 8 now interact with the RAG layer.

1

HealthAgent

Uses Polars for performance. Checks completeness, duplicates, consistency (IQR outliers >5%). Computes Trust Score (0–100): completeness 40pts + duplicates 30pts + consistency 30pts.

2

DataAgent

Strips whitespace, drops 100% NaN rows/cols, coerces object→numeric (≥70% parse). Mean-imputes numerics, mode-imputes categoricals. Smart encoding: drops ID-like cols (>95% unique), label-encodes high-cardinality (>50), one-hot encodes low-cardinality.

3

SchemaInsightAgent RAG Creator Groq AI

Creates the RAGRetriever: RAGRetriever(df) — if dataset has >20 columns, indexes all columns into ChromaDB using MiniLM embeddings and stores it in context["rag_retriever"] for reuse. Then detects column semantics (date, monetary, rating, etc.), generates data-specific Plotly charts, and asks Groq for 5 business questions using RAG-retrieved schema (not just the first 20 columns).

4

TargetAgent RAG Consumer Groq AI

Uses RAGRetriever from context if available. Queries: "prediction target outcome binary classification label churn fraud" → retrieves the most target-like column names instead of blindly slicing columns[:15]. This means a churn_flag column at index 45 is now surfaced correctly. Detects data leakage (>0.95 correlation with target).

5

FeatureAgent

Skipped if skip_ml=True. Correlation heatmap (Matplotlib/Seaborn) for top 10 features vs target.

6

ModelAgent

Samples to 2000 rows, applies StandardScaler. Classification: Logistic Regression + Random Forest + Gradient Boosting. Regression: Linear + Ridge + Random Forest + Gradient Boosting. Produces Plotly JSON bar chart. Flags leakage if accuracy > 97%.

7

EvaluationAgent

Target distribution plot, confusion matrix heatmap, ROC curve + AUC (binary classification). All saved as PNGs in outputs/.

8

InsightAgent RAG Consumer Groq AI

Uses RAGRetriever from context. Queries: "key metrics business outcome important factors performance indicators" — semantically different from TargetAgent's query, so retrieves a different column subset on wide datasets. Generates executive summary + 5 recommendations + 2 discovery insights (ML mode) or EDA business intelligence narrative (EDA mode).

9

AdvancedInsightsAgent

Temporal trends (MoM growth), anomaly detection (3σ Z-scores), sensitivity/what-if analysis (correlation-based projections). Uses Pandas + Polars.

10

ReportAgent

Multi-page ReportLab PDF: branded header, exec summary, describe table, model comparison, visual insights (heatmap, confusion matrix, ROC), recommendations.

RAG Layer NEW

InsightForge uses a threshold-aware Retrieval-Augmented Generation system to improve LLM prompt quality on wide datasets. It does NOT use RAG for small datasets — this is a deliberate design decision.

The Core Rule — One Constant Controls Everything
# data_engine/vector_store.py
COLUMN_RAG_THRESHOLD = 20   # change once → affects all agents + QueryAgent
Dataset SizeRAG Active?BehaviourOverhead
≤ 20 columns❌ OFFFull schema sent directly to LLM — same as beforeZero
> 20 columns✅ ONQuestion embedded → top-k columns retrieved → focused prompt~15ms per embed

Architecture — 3 Files, Clean Separation

vector_store.py — SessionVectorStore
  • ChromaDB EphemeralClient — purely in-memory, zero disk writes, auto-GC'd when the object is dropped
  • Process-level model singleton — _get_model() loads all-MiniLM-L6-v2 once per server process (not per request). "Loading weights 103/103" appears exactly once at startup.
  • Column document format: "Column: {name} | Type: {dtype} | Samples: {top3_values}" — sample values give MiniLM semantic signal beyond the column name (e.g. "Male, Female, Non-binary" identifies a gender column even if named col_42)
  • Cosine similarity ({"hnsw:space": "cosine"}) for the ChromaDB collection
rag_retriever.py — RAGRetriever

The single wrapper that all agents and QueryAgent use. Hides the threshold logic so agents never check column counts themselves.

retriever = RAGRetriever(df)                   # checks threshold in __init__
schema = retriever.get_schema(question, k=12)  # RAG or full — same API call
  • get_schema(question, k) — returns formatted schema text for LLM prompt injection
  • get_column_names(question, k) — returns list of column names for custom use
  • Each agent uses a semantically different query string to retrieve domain-appropriate columns

Why Not Always Use RAG?

For datasets with ≤ 20 columns, the full schema is short enough to fit in any LLM context window. Adding RAG would mean: loading ChromaDB + running MiniLM embeddings (~15–50ms) to retrieve... all 20 columns anyway. That's pure overhead with zero quality gain. The threshold is the right engineering call.

Chat-with-Data Engine

QueryAgent — 4-Tier Architecture (updated)
  1. Geo Synonym Injection (pre-processing): Detects regional aliases in the question ("Pacific coast", "Midwest", "New England"), resolves them to exact column values for THIS dataset, injects a "fact sheet" into the system prompt before the LLM writes SQL. Always runs — returns "" if no alias detected (zero overhead).
  2. Tier 1 — RAG + Groq AI (best quality): If dataset > 20 cols, embeds question → retrieves top-12 relevant column names → builds focused schema → sends to Groq → Groq generates DuckDB SQL → executes → Groq interprets result.
  3. Tier 2 — Local Pattern Matching: Regex detects intent (top/best, average, sum, distribution, correlation, what-if, show data) → generates SQL locally → DuckDB → formatted markdown table. No API call.
  4. Tier 3 — Smart Fallback: Finds closest column via word-overlap scoring → returns stats for that column. Never returns empty.

SQL Sanitisation

# After Groq responds, before DuckDB execution:
sql = sql.replace("```sql", "").replace("```", "").strip()
# DuckDB uses "Column" (double-quote), not `Column` (backtick/MySQL).
# Llama occasionally outputs MySQL-style backticks — this converts them:
sql = re.sub(r'`([^`]+)`', r'"\1"', sql)

Geographic Synonym Layer NEW

Problem Solved

When a user asks "sales from the Pacific coast", the LLM would generate WHERE Region = 'Pacific' — but the Superstore dataset has WHERE "State" IN ('California', 'Oregon', ...). The SQL ran successfully but returned 0 rows → sum() = NaN. The geo synonym layer solves this permanently.

Three-Step Implementation

Step 1 — Static dictionary (module-level, _GEO_SYNONYMS): 20 aliases covering all US Census Bureau divisions. Keys are lowercase aliases; values are full state names.

"pacific coast":    ["California", "Oregon", "Washington", "Alaska", "Hawaii"],
"pacific division": ["California", "Oregon", "Washington", "Alaska", "Hawaii"],
"midwest":          ["Illinois", "Indiana", "Iowa", "Kansas", "Michigan", ...],
"new england":      ["Connecticut", "Maine", "Massachusetts", ...],
"east coast":       ["Maine", "New Hampshire", ..., "Florida"],
# + 15 more divisions/regions

Step 2 — Format detection at init (_detect_geo_column()): Runs once per QueryAgent instantiation (one DuckDB DISTINCT query). Finds the geographic column by scanning column names for "state", "province", "region", etc. Then checks 30 sample values:

If values look like...Detected formatExpansion used
"California", "Texas"fullFull state names
"CA", "TX"abbrev2-letter codes (via _STATE_ABBREVS)
"West", "South"regionFull state names (LLM matches via sample data)

Step 3 — System prompt injection (_build_geo_context(question)): Called per question. Returns "" if no alias detected (zero overhead for non-geographic questions). When an alias IS detected, injects this before the LLM writes SQL:

Geographic synonym mappings for this dataset:
 - 'pacific coast' → "State" IN ('California', 'Oregon', 'Washington', 'Alaska', 'Hawaii')
Use these exact column values when writing your SQL WHERE clause.

Result for the failing query

# Before: Groq wrote this → returned NaN
SELECT SUM(`Sales`) FROM data WHERE "Region" = 'Pacific'

# After: Groq receives the synonym hint → writes correct SQL
SELECT SUM("Sales") FROM data
WHERE "State" IN ('California', 'Oregon', 'Washington', 'Alaska', 'Hawaii')
-- Returns: actual number ✅

Async Job System

Background Thread Execution
  1. dispatch_pipeline(df, analysis_id) → sets status "queued" → starts threading.Thread(daemon=True)
  2. Thread runs PipelineCoordinator.run(df, on_step=callback)
  3. Each agent step → callback updates Redis: {"status": "running", "step": "ModelAgent", "progress": 0.6}
  4. On completion → status="done", results in _results dict
  5. Frontend polls GET /jobs/{id}/status → progress bar
  6. GET /jobs/{id}/result pops results (one-time fetch) → persists to SQLite

Interview Q&A

Walk me through what happens when a user uploads a CSV.
POST multipart to /analyze. parse_upload() detects extension, uses charset_normalizer to detect encoding, reads into Pandas DataFrame. Saved as Parquet for chat/RAG. If authenticated, an AnalysisSession row is created in SQLite. dispatch_pipeline() launches a daemon thread and returns job_id immediately. Frontend polls GET /jobs/{id}/status (Redis) for progress. When done, GET /jobs/{id}/result returns the full JSON payload which is also persisted to SQLite.
What is RAG and how do you use it in InsightForge?
RAG = Retrieval-Augmented Generation. Normally you'd embed documents and retrieve relevant chunks before sending to an LLM. Here I apply the same idea to dataset columns: instead of dumping all 60+ column names into a Groq prompt, I embed the user's question with MiniLM (80MB model), query a ChromaDB index of embedded column descriptions, retrieve the top-12 most relevant columns, and send only those. This gives the LLM a focused, accurate schema — fewer hallucinated column names, faster API calls, and correct SQL on wide datasets. The key design decision is the threshold: RAG only activates for datasets with more than 20 columns. Below that, the full schema is short enough to send directly — adding RAG would just be overhead.
Why use an ephemeral ChromaDB instead of a persistent vector store?
The column index only needs to exist for the duration of a chat session. The actual dataset is already persisted as a Parquet file. Using EphemeralClient() means no disk I/O, no port to manage, and the index is automatically garbage-collected when the QueryAgent object drops. If we wanted persistence, we'd just re-index from the Parquet file — it takes ~100ms for 200 columns. There's no value in persisting an index that can be rebuilt instantly from the source data.
How does the geographic synonym layer work?
Three steps. First, a static dictionary maps 20 US Census Bureau division aliases to lists of state names — authored once, never changes. Second, when QueryAgent initializes, it runs one DuckDB DISTINCT query to sample 30 values from the geographic column and determines if the dataset uses full names ("California"), abbreviations ("CA"), or region labels ("West"). Third, before each Groq SQL call, _build_geo_context(question) scans the question for any known alias. If "pacific coast" is found, it injects a paragraph into the system prompt: "'pacific coast' → State IN ('California', 'Oregon', 'Washington', 'Alaska', 'Hawaii')". The LLM reads this and uses the exact IN clause. For non-geographic questions, the function returns an empty string — zero overhead.
What was the Pacific coast NaN bug and how did you fix it?
Two bugs in one error. Bug 1: the LLM wrote WHERE Region = 'Pacific' but the Superstore dataset has a State column with full state names like "California", not a Region column with "Pacific". The SQL ran, returned 0 rows, and SUM() of an empty set is NULL/NaN. Fix: geographic synonym injection so the LLM writes WHERE State IN ('California', ...). Bug 2: Llama-3.1 sometimes outputs MySQL-style backtick quoting — SUM(`Sales`) — which DuckDB cannot parse (it throws "Scalar Function __postfix does not exist"). Fix: one regex line after receiving the LLM response: sql = re.sub(r'`([^`]+)`', r'"\\1"', sql).
Why does the SentenceTransformer model show "Loading weights 103/103" on startup?
The all-MiniLM-L6-v2 model is ~80MB and stored in the local HuggingFace cache after first download. The progress bar is from sentence_transformers loading the 103 weight shards from disk. The original bug was that SentenceTransformer("all-MiniLM-L6-v2") was being instantiated inside SessionVectorStore.__init__() — meaning it reloaded from disk on every chat request. The fix: a module-level singleton _MODEL with a _get_model() lazy loader. Now it loads once per server process, and all SessionVectorStore instances share the same object. The "unauthenticated HF Hub" warning is suppressed via HF_HUB_DISABLE_IMPLICIT_TOKEN=1 in .env.
How does the pipeline decide ML vs EDA?
TargetAgent (step 4) makes the decision. First checks keyword matches (churn, fraud, survived…) across all column names. If none match, uses RAGRetriever (if >20 cols) to retrieve the most target-like columns, then asks Groq AI: "Is there a meaningful prediction target?" with schema + sample. If the AI returns a valid column that passes quality checks (2–20 unique values, not an ID, <80% unique ratio), ML proceeds. Otherwise skip_ml=True is set in the context dict and agents 5–7 skip. InsightAgent then switches to EDA-only narrative mode.
What's your scaling strategy?
Incremental. Step 1: SQLite → PostgreSQL (change DATABASE_URL, SQLAlchemy abstracts it). Step 2: threads → Celery workers (already in requirements.txt). Step 3: Parquet files → S3/GCS with a storage abstraction. Step 4: FastAPI is stateless (JWT auth) so it scales horizontally once the in-memory _results dict moves to Redis. The RAG layer scales naturally — each ChromaDB EphemeralClient is per-request and in-memory, so horizontal scaling just means each instance has its own index (which is fine, re-indexing from Parquet takes ~100ms).

How to Explain InsightForge in 2 Minutes

"InsightForge is a full-stack data intelligence platform.

The problem: Non-technical teams waste hours in Excel. Data scientists spend 80% of their time on cleaning and setup, not insights.

Core flow: You upload any dataset — CSV, Excel, SQLite, even ZIP files — and a 10-agent AI pipeline automatically cleans the data, detects whether ML is appropriate, trains multiple models, generates interactive Plotly charts, and writes an executive summary with business recommendations. It produces a PDF report you can hand to your manager, plus PPTX and Excel exports.

The smart part: A TargetAgent uses Groq AI to evaluate whether the data has a meaningful prediction target. If it's just a customer list or transaction log, it switches to pure EDA mode — business intelligence insights instead of garbage predictions.

Chat with your data: Ask natural language questions like "What are Pacific coast sales?" The system has a geographic synonym layer that knows "Pacific coast" = California, Oregon, Washington, Alaska, Hawaii — it injects this into the LLM prompt so the SQL is always correct. For wide datasets (50+ columns), we use a RAG layer: embed the question with MiniLM, retrieve only the relevant columns from ChromaDB, send a focused schema to the LLM — faster and more accurate than dumping 60 columns into the prompt.

Tech stack: FastAPI + React 19 + Vite, SQLite (swappable to PostgreSQL), Redis caching with in-memory fallback, JWT auth, APScheduler for recurring reports, ChromaDB + all-MiniLM-L6-v2 for RAG, DuckDB for in-process analytical SQL. Runs as a single Docker container."
